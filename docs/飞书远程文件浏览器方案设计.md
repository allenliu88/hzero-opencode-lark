# 飞书远程文件浏览器方案设计

## 1. 背景

`opencode-lark` 已经可以把飞书消息发送到 OpenCode session，并将 Agent 回复显示为飞书卡片。用户也可以要求 Agent 调用 `read` 工具读取文件，但该方式需要调用模型，不适合频繁浏览目录和查看原始文件内容。

OpenCode Server 提供只读文件接口：

```http
GET /file?path=<relative-path>&directory=<workspace-directory>
GET /file/content?path=<relative-path>&directory=<workspace-directory>
```

其中 `/file` 用于列出目录，`/file/content` 用于读取文件内容。文件操作发生在 OpenCode Server 所在主机，不是 `opencode-lark` 所在主机。因此，即使两者分离部署，也可以由桥接服务通过 HTTP 获取远端项目文件并展示到飞书。

当前工程已经具备以下基础能力：

- `SessionManager` 维护飞书 thread 到 OpenCode session 的绑定。
- `command-handler.ts` 支持 `/sessions`、`/connect` 等命令和交互卡片。
- `ws-client.ts` 接收 `card.action.trigger`，并在 3 秒内返回回调响应。
- `index.ts` 统一分发卡片动作。
- `AgentConsoleRegistry` 已验证“按卡片消息 ID 保存服务端交互状态”的模式。
- `FeishuApiClient.updateMessage()` 可以更新已有普通交互卡片。

本方案在这些能力之上增加一个只读的远程文件浏览器。

## 2. 设计目标

- 用户可从飞书打开当前绑定 OpenCode session 的项目目录。
- 用户可在同一张卡片中逐层进入目录、返回上级和回到根目录。
- 用户可查看文本文件，并按行分页浏览。
- 浏览目录和读取文件不调用模型，不产生模型 Token 消耗。
- 文件始终从 OpenCode Server 读取，支持桥接服务与 OpenCode 分离部署。
- 卡片回调快速确认，远端请求和卡片更新在后台完成。
- 不信任卡片回调中的路径、session ID、directory 或用户身份。
- 同一张卡片的并发点击不会发生旧响应覆盖新视图。
- 所有注册、队列和缓存都具有明确的 TTL 和释放路径。

## 3. 非目标

- 第一阶段不支持编辑、创建、删除、移动或重命名文件。
- 第一阶段不提供完整 IDE 能力，如语法高亮、符号跳转和差异编辑。
- 不一次性递归加载完整目录树。
- 不读取当前 OpenCode session 工作目录之外的文件。
- 不通过 `opencode-lark` 本机文件系统读取远端项目文件。
- 不用 Agent 的 `read` 工具作为文件 API 失败后的自动降级。
- 第一阶段不持久化浏览器状态；服务重启后旧卡片失效。
- 第一阶段不直接预览二进制、图片、音视频或压缩包内容。

## 4. 用户入口

新增命令：

```text
/files
/files src
```

规则：

- `/files` 打开当前绑定 session 的项目根目录。
- `/files <relative-path>` 打开根目录下的指定目录或文本文件。
- 只读命令使用 `SessionManager.getExisting()`，没有绑定时不自动创建 session。
- 群聊 thread 沿用现有 `feishuKey` 规则，确保浏览的是当前话题绑定的 session。
- `/help` 增加“浏览文件”入口，但目录浏览动作使用独立 action type，不复用一次性 `command_execute` 锁定逻辑。

未绑定 session 时返回：

```text
当前话题尚未连接 OpenCode 会话，请先使用 /sessions 选择会话。
```

## 5. 目标交互

### 5.1 目录视图

飞书卡片不适合无限展开的传统树形控件。目录采用“面包屑 + 当前层列表”的懒加载方式：

```text
远程文件浏览器

会话  ses_0a15227e6ffeV837tzQzDxPBjb
路径  /src/handler

📁  __tests__
📁  adapters
📄  command-handler.ts
📄  message-handler.ts
📄  outbound-media.ts

[上一级] [根目录] [刷新] [上一页] [下一页]
第 1/3 页
```

交互规则：

- 点击目录后在原卡片中显示该目录的直接子项。
- 点击文件后切换到文件内容视图。
- 目录项使用 Card JSON 2.0 单列全宽文本按钮，点击图标或名称均可进入；不额外显示右侧箭头。
- 目录使用 `📁`，文件使用 `📄`，不展示 `[目录]`、`[文件]` 文本前缀。
- 完整展示 session ID，不按固定字符数截断。
- 目录优先于文件，名称按不区分大小写的字典序排列。
- 每页默认显示 8 项，具体值作为模块常量管理。
- 当前路径通过文本面包屑展示；第一阶段面包屑不逐段点击。
- “上一级”在根目录禁用或不渲染。
- “上一页”和“下一页”只在对应方向存在页面时渲染。
- “刷新”重新请求当前目录，并保持当前页；若总页数缩小则回到最后一页。
- 上一级、根目录、刷新、上一页和下一页合并在一个 `flex_mode: "none"` 的单行操作区，不拆分为多行。

### 5.2 文件视图

文本文件按行分页显示：

````text
远程文件浏览器

src/handler/command-handler.ts
第 1-60 行，共 460 行

```ts
1  import type { SessionManager } from "../session/session-manager.js"
2  ...
```

[返回目录] [刷新] [上一页] [下一页]
````

交互规则：

- 默认每页最多 60 行。
- 同时受 UTF-8 字节预算约束，避免卡片超过平台限制。
- 代码围栏语言由扩展名映射，未知扩展名使用 `text`。
- 文件内容中的三反引号需要转义或改用安全渲染策略，保证卡片 Markdown 结构完整。
- “返回目录”回到打开该文件前的目录和页码。
- “刷新”重新读取文件；文件变短时修正当前页。
- 文件页所有操作也固定在同一个单行操作区。
- 第一阶段不增加“跳转到指定行”表单，后续可按真实使用需求补充。

### 5.3 不可预览文件

满足以下任一条件时不渲染文件正文：

- OpenCode API 将内容标记为二进制。
- 内容无法按支持的文本响应解析。
- 文件超过预览总量上限。
- 文件名命中敏感文件策略。

卡片显示不可预览原因并提供根目录和重试操作。第一阶段不提供远端文件下载，因为现有 `outbound-media.ts` 只能读取桥接服务本机路径，无法安全上传 OpenCode Server 上的远端文件。

## 6. OpenCode 数据契约

### 6.1 解析当前 workspace

文件浏览器先读取当前绑定：

```text
feishuKey -> SessionManager.getExisting(feishuKey) -> sessionId
```

随后请求：

```http
GET /session/{sessionId}
```

从 session 响应取得服务端记录的 `directory`。该字段是本次浏览器实例的根目录，也是后续文件 API 的 workspace routing 参数。

关键约束：

- 不接受飞书命令或回调传入绝对 `directory`。
- 每次创建浏览器时固定 `sessionId` 和 `directory` 快照。
- 浏览期间飞书 thread 改绑其他 session，不会静默改变旧卡片的根目录。
- 可选增强是在每次回调前重新校验 session 仍存在；第一阶段仅在 OpenCode 返回 404 时将浏览器置为失效状态。

### 6.2 列目录

请求：

```http
GET /file?path=<relative-path>&directory=<workspace-directory>
```

适配层将 OpenCode 原始响应归一化为内部模型：

```ts
interface RemoteFileEntry {
  name: string
  path: string
  type: "file" | "directory"
  size?: number
}
```

`RemoteFileClient` 负责隔离 OpenCode SDK 或 HTTP API 的具体响应结构。卡片层不能直接依赖上游原始 JSON。

### 6.3 读文件

请求：

```http
GET /file/content?path=<relative-path>&directory=<workspace-directory>
```

适配层归一化为：

```ts
interface RemoteFileContent {
  path: string
  content: string
  encoding: "utf-8"
  sizeBytes: number
}
```

实现前需要用当前锁定的 `@opencode-ai/sdk` 版本或目标 OpenCode Server 进行契约探测，确认目录项字段、文件内容字段、错误码和大文件行为。若 SDK 已稳定覆盖 `file.list/read`，优先扩展启动时创建的 SDK client；否则使用集中封装的 HTTP client，禁止在 handler 中散落 `fetch()` 和响应断言。

### 6.4 鉴权和超时

若 OpenCode Server 设置了 `OPENCODE_SERVER_PASSWORD`，文件 client 必须与其他 OpenCode 请求共享 Basic Authentication 配置。该能力应通过统一的 OpenCode client 或请求封装提供，不能只给文件浏览器单独实现一套凭证逻辑。

当前限制：

- session 查询超时 5 秒。
- 目录查询超时 5 秒。
- 文件查询超时 10 秒。
- 文件内容响应上限 1 MiB；达到上限后终止读取并显示不可预览。
- 不将凭证、完整响应正文或文件内容写入日志。

## 7. 卡片状态模型

每张浏览器卡片对应一个服务端状态：

```ts
interface FileBrowserSession {
  messageId: string
  chatId: string
  operatorOpenId: string
  sessionId: string
  directory: string
  view:
    | {
        type: "directory"
        path: string
        page: number
      }
    | {
        type: "file"
        path: string
        page: number
        returnPath: string
        returnPage: number
      }
  viewToken: string
  entries: Map<string, RemoteFileEntry>
  pendingActionIds: Set<string>
  queue: Promise<void>
}
```

状态由 `FileBrowserRegistry` 管理，以飞书卡片 `open_message_id` 为主键：

```ts
interface FileBrowserRegistry {
  enqueue(action: FeishuCardAction): FileBrowserActionResult
  unregister(messageId: string): void
  close(): void
}
```

默认 TTL 为 30 分钟，用户每次成功操作后续期。TTL 到期、服务关闭或明确失效时释放状态和 timer。

## 8. 回调协议

目录项按钮只携带动作、当前视图令牌、不透明 entry key 和幂等动作 ID：

```json
{
  "action": "file_browser_open_entry",
  "viewToken": "7f9a...",
  "entryKey": "7f9a...:2",
  "actionId": "7f9a...:file_browser_open_entry:7f9a...:2"
}
```

导航按钮示例：

```json
{ "action": "file_browser_parent" }
{ "action": "file_browser_root" }
{ "action": "file_browser_refresh" }
{ "action": "file_browser_page", "direction": "next" }
{ "action": "file_browser_back" }
```

`entryKey` 到实际路径的映射保存在当前服务端渲染快照中。`viewToken` 防止旧卡片按钮命中新视图，`actionId` 防止飞书重试或用户双击导致重复执行。回调中不接受以下权威字段：

- 绝对路径；
- workspace directory；
- OpenCode session ID；
- 任意用户指定的页码；
- 文件类型。

即使用户伪造 callback value，也只能触发注册表中当前卡片允许的有限动作。

## 9. 回调处理与卡片更新

飞书要求卡片回调在约 3 秒内完成。文件浏览器在同步阶段完成 message、chat、operator、viewToken、entryKey 和 actionId 校验，将已解析的不可变动作入队后立即返回准确 toast：

```text
card.action.trigger
  -> FileBrowserRegistry 同步校验并解析动作
  -> 立即返回加载、重复、过期或无权限 toast
  -> 原卡显示轻量加载态
  -> 后台请求 OpenCode
  -> 构建新卡片 JSON
  -> updateMessage 更新原卡片
```

WebSocket 和 HTTP webhook 都支持同步返回文件浏览预检结果；其他耗时卡片处理器仍保持 fire-and-forget。合法动作显示“正在打开目录/文件”，重复动作、旧视图、过期状态和非创建者操作返回不同提示。

目录浏览使用 Card JSON 2.0 普通交互卡，不是 streaming card。每次导航通过 `FeishuApiClient.updateMessage()` 替换原卡片，原因如下：

- 目录项和按钮结构每次都会变化，整卡更新比动态增删元素更简单。
- 浏览器不存在高频 SSE 更新，不需要 CardKit streaming 调度器。
- 可以复用现有命令菜单的普通卡片能力。

## 10. 并发与顺序保证

用户可能快速点击多个按钮，飞书也可能重试回调。每张卡片具备独立串行 mutation 队列、view token 和 action ID 集合：

1. 同步校验当前 `viewToken`。
2. 入队前将 entry key 解析为不可变目标动作。
3. 用 `actionId` 拒绝重复执行。
4. 合法动作入队后立即使旧 `viewToken` 失效。
5. 同卡动作串行执行，单次失败被捕获后队列继续可用。
6. 不同卡片可以并发请求。

OpenCode session/目录请求分别使用 5 秒超时，文件读取使用 10 秒超时。超时通过 AbortSignal 中止并释放队列。飞书卡片更新最多尝试 3 次，使用 200ms、600ms 短退避；加载态更新失败不阻塞实际目录或文件请求。

目录页和文件页都使用服务端状态计算上一页、下一页，不信任 callback 中的任意页码。

## 11. 安全设计

### 11.1 信任边界

可信数据：

- `SessionManager` 返回的当前绑定 session ID；
- OpenCode `/session/{id}` 返回的 `directory`；
- OpenCode 文件 API 返回的目录项和文件内容；
- 注册表中由服务端生成的 entry key 映射。

不可信数据：

- 飞书命令参数；
- 卡片 callback value；
- `open_message_id`、`open_chat_id` 和 operator 的组合；
- 文件名和文件内容本身；
- OpenCode API 的异常或不完整响应。

### 11.2 路径约束

客户端先执行纯词法校验：

- 将 `\\` 统一为 `/` 或直接拒绝反斜杠路径；
- 拒绝绝对路径；
- 拒绝空字节；
- 规范化 `.`；
- 拒绝规范化后仍包含 `..` 的路径逃逸；
- 根目录内部统一用 `.` 或空相对路径表示。

桥接服务与 OpenCode 可能不在同一主机，因此不能用桥接服务本机的 `fs.realpath()` 验证远端路径。最终文件系统边界必须由 OpenCode Server 的 workspace-scoped 文件 API保证。若目标 OpenCode 版本允许 `/file` 通过符号链接逃出 workspace，则该版本不满足本功能的安全前提，应禁用文件浏览器，而不是在桥接服务端伪造本机校验。

### 11.3 操作者约束

默认只有发起 `/files` 的用户可以操作该浏览器卡片：

- 注册时保存发起者 `operatorOpenId`；
- 回调时校验 `action.operator.open_id`；
- 同时校验 `open_chat_id` 和 `open_message_id`；
- 非发起者点击时不执行远端请求，并记录不含敏感路径的拒绝原因。

当前 `CommandHandler` 未接收 sender open ID。实施时需要扩展命令上下文，而不是从不可靠字段推断发起者。

### 11.4 敏感文件策略

即使文件位于项目根目录，也可能包含密钥。默认拒绝预览以下文件或模式：

```text
.env
.env.*
*.pem
*.key
id_rsa
id_ed25519
*.p12
*.pfx
```

目录中可以隐藏这些条目，或者显示为“受保护文件”但不可点击。第一阶段建议隐藏并记录不含路径详情的审计日志。敏感规则应集中定义并可配置扩展，但不提供关闭默认规则的兼容开关。

### 11.5 内容安全

- 文件内容按不可信 Markdown 处理。
- 防止内容中的三反引号破坏代码围栏。
- 不把文件内容插入按钮 value、URL 或日志。
- 错误卡不展示 OpenCode 原始响应正文、认证信息或服务器绝对路径。
- 卡片只展示相对于 workspace 的路径。

## 12. 错误状态

错误在原卡片内展示，并保留可恢复操作：

| 场景 | 用户提示 | 可用操作 |
| --- | --- | --- |
| 未绑定 session | 当前话题尚未连接会话 | 打开 `/sessions` |
| session 不存在 | 会话已失效 | 重新选择会话 |
| OpenCode 不可达 | 无法连接 OpenCode 服务器 | 重试 |
| 目录不存在 | 目录已不存在 | 返回上级、根目录 |
| 文件不存在 | 文件已不存在 | 返回目录 |
| 权限不足 | 无权读取该路径 | 返回目录 |
| 敏感文件 | 该文件不允许预览 | 返回目录 |
| 二进制文件 | 该文件不支持文本预览 | 返回目录 |
| 文件过大 | 文件超过预览上限 | 返回目录 |
| 浏览器过期 | 浏览会话已过期 | 重新发送 `/files` |

错误更新失败时只记录结构化错误，不循环重试卡片更新。

## 13. 模块划分

建议新增：

```text
src/file-browser/
  remote-file-client.ts       OpenCode 文件 API 适配和响应归一化
  path-policy.ts              相对路径和敏感文件策略
  file-browser-card.ts        目录、文件、加载和错误卡片构建
  file-browser-registry.ts    卡片状态、entry key、TTL、串行操作
  types.ts                    内部模型
```

现有文件改动：

- `src/handler/command-handler.ts`：增加 `/files`，接收命令上下文并创建浏览器。
- `src/handler/message-handler.ts`：向命令处理器传递 sender open ID 和精确 thread key。
- `src/index.ts`：实例化 file client/registry，分发 `file_browser_*` action，关闭时释放注册表。
- `src/feishu/interactive-card-response.ts`：为文件浏览动作返回即时 toast。
- `src/types.ts` 或命令模块类型：补充命令上下文类型。
- `README.md`、`README.zh-CN.md`：记录 `/files` 用法和安全限制。

文件浏览逻辑不放入 `interactive-handler.ts`。该模块当前专注 OpenCode question 和 permission 回复；文件浏览有独立状态和生命周期，应由独立 registry 处理。

## 14. 可观测性

建议记录以下结构化字段，但不记录文件内容：

```text
action=file_browser_open
message_id
chat_id
operator_id_hash
session_id
relative_path_hash
entry_type
duration_ms
result=success|not_found|forbidden|timeout|invalid_response
```

建议指标：

- 目录请求成功率和 p95 延迟；
- 文件请求成功率和 p95 延迟；
- 卡片更新成功率；
- 回调被拒绝次数；
- 敏感文件拦截次数；
- 浏览器注册表当前数量和 TTL 回收数；
- 同卡队列等待时长。

第一阶段只要求结构化日志，指标系统接入不作为上线阻塞项。

## 15. 兼容与降级

- 不改变现有普通消息、Agent 文件读取和 outbound media 链路。
- `/files` 是新增命令，不影响未知 slash command 继续交给 Agent。
- OpenCode 版本不支持文件 API 时，返回明确“不支持远程文件浏览”，不自动调用模型。
- CardKit callback 未配置时，初始目录卡可显示，但点击会失败；README 必须继续强调 `card.action.trigger` 配置。
- 服务重启后旧浏览器卡片点击返回“浏览会话已过期”，用户重新发送 `/files` 即可。

## 16. 验证范围

- `/files` 使用当前 thread 已绑定的 session，不自动创建 session。
- 根目录和指定相对路径可以打开。
- 目录懒加载、排序和分页正确。
- 文本文件按行和字节预算分页。
- 返回目录、上级、根目录、刷新、上一页和下一页正确。
- callback 校验 message、chat、operator 和 entry key。
- 伪造绝对路径、`..`、session ID、directory 和页码无效。
- 敏感文件不可见或不可打开。
- 二进制、超大、消失和无权限文件有明确错误态。
- 同卡快速连续操作保持顺序，失败后队列可继续使用。
- 不同浏览器卡片可以并发。
- TTL、服务关闭和失效路径释放 timer 与状态。
- OpenCode 不可达、超时、非 JSON 和异常响应不会泄漏敏感信息。
- 桥接服务与 OpenCode 分离部署时仍从 OpenCode 主机读取文件。

## 17. 决策摘要

- 使用 `/files` 作为入口。
- 使用面包屑和当前层懒加载，不渲染完整递归树。
- 目录调用 `/file`，文件调用 `/file/content`。
- workspace 根目录来自绑定 session 的 `directory`。
- 使用 Card JSON 2.0 普通交互卡整卡更新，不使用 streaming card。
- 目录项使用单列全宽图标按钮；底部全部操作固定为单行。
- 按卡片消息 ID 保存服务端状态，callback 携带有限动作、view token、不透明 entry key 和 action ID。
- 默认仅命令发起者可操作。
- 第一阶段只读、只预览受支持文本，不提供远端下载。
- 默认拦截常见凭证文件。
- 远端真实路径边界依赖 OpenCode workspace-scoped 文件 API；不使用桥接主机的 `realpath()` 冒充远端校验。
