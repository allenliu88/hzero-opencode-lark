# 飞码智能体流式卡片优化实施计划

## 1. 计划目标

本计划用于落实 `docs/飞码智能体流式卡片优化方案设计.md`，分阶段将当前“工具进度卡 + 最终回复卡”升级为稳定、低延迟、可交互的飞码智能体会话面板。

实施原则：

- 先修正确性与稳定性，再调整产品结构。
- 每个阶段都必须可独立发布和回滚。
- 不以高频请求换取视觉动效。
- 性能优化必须用请求数、延迟和泄漏指标验收。
- 问题和权限可靠链路优先于单卡视觉统一。

## 4. 阶段 2：双元素主卡与答案流式展示

### 4.1 目标

将主卡从单个 `content` markdown 升级为：

```text
progress element
answer element
```

`TextDelta` 经 300ms latest-only 调度器更新 `answer`，`ToolStateChange` 更新 `progress`。正常完成后关闭同一张卡，不再额外发送最终回复卡。

### 4.2 数据结构

新增或抽取：

```ts
interface CardUpdateSnapshot {
  progress?: string
  answer?: string
}

class CardUpdateScheduler {
  updateProgress(content: string): void
  updateAnswer(content: string): void
  flush(): Promise<void>
  close(): Promise<void>
}
```

要求：

- 整张卡共用一个 sequence。
- 每个 element 记录 lastSentContent。
- pending 只保留每个 element 的最新内容。
- 多 element 更新按固定顺序执行。
- close 后拒绝普通更新。

- `SessionIdle` 后 2 秒内主卡稳定为完成态。
- 同一请求正常情况下只产生一张主结果卡。
- 卡片内容不超过安全字节预算。
- 当前执行动作后可看到内联耗时，例如 `正在执行命令 bun test · 已用时 8 秒`。
- 工具结束后显示准确的最终耗时。
- 长任务计时频率会自动降低，不保持每秒请求。
- 计时更新不会延迟工具状态、答案或最终完成状态。

### 7.3 压力场景

| 场景 | 输入 | 目标 |
| --- | --- | --- |
| 快速纯文本 | 100 delta / 2 秒 | 不逐 delta 请求，答案正确 |
| 工具突发 | 1000 状态事件 / 1 秒 | update 数受控，最终状态正确 |
| 慢 CardKit | RTT 500ms | pending <= 1，不积压历史快照 |
| 429 | 连续两次后成功 | 自动退避恢复 |
| 更新失败 | 首次 update reject | 后续更新和 close 成功 |
| 丢失 idle | 首事件后静默 | watchdog 清理全部资源 |
| 同 session 并发 | 两个不同答案 | 不串流、不重复 |
| 长任务 | 60 分钟 | heartbeat 正常，无泄漏 |
| 等待权限 | 30 分钟后处理 | 不误超时，恢复正常 |

## 8. 发布策略

建议增加配置或 feature flag：

```text
streaming.console.enabled
streaming.answer.enabled
streaming.updateIntervalMs
streaming.cardDelayMs
streaming.inactivityTimeoutMs
streaming.waitingTimeoutMs
streaming.maxLifetimeMs
```

发布顺序：

1. 开发环境启用双元素答案流式。
2. 小范围群聊灰度。
3. 监控 429、请求耗时、更新合并率和最终响应延迟。
4. 指标稳定后扩大范围。
5. 保留普通最终回复卡作为故障兜底，而不是长期双发。

回滚条件：

- 429 显著增加。
- `SessionIdle -> final visible` p95 超过 3 秒。
- 卡片关闭失败持续升高。
- 出现答案丢失或跨请求串流。
- active listener 或 timer 数量持续增长。

## 9. Definition Of Done

每个阶段完成前必须满足：

- 相关单元测试和集成测试通过。
- `bun run build` 通过。
- `bun run test:run` 通过。
- 静态检查无新增问题。
- 没有未清理 timer、listener 和 request lock。
- 错误路径有结构化日志。
- 关键指标可以观测。
- 文档同步更新阶段状态和参数。
- 真机验证桌面端和至少一个移动端。

## 10. 推荐执行顺序

1. 补齐阶段 1 剩余故障测试。
2. 实现双元素 CardKit schema 和统一 scheduler。
3. 接入 `TextDelta` 真流式答案。
4. 实现短任务延迟建卡和长内容降级。
5. 增加同 session 单飞保护。
6. 扩展 message 级事件归属。
7. 实现继续对话和关闭会话。
8. 建设指标、压力测试和真机参数调优。

该顺序可以先获得用户可感知的答案流式体验，同时避免在请求归属和稳定性尚未验证时一次性引入过多交互复杂度。
