# 飞码智能体流式卡片优化方案设计

## 1. 背景

`opencode-lark` 负责将飞书消息投递到 OpenCode session，并将 SSE 事件转换为飞书卡片和消息。当前工程已经具备以下基础能力：

- `EventProcessor` 将原始 SSE 转换为 `TextDelta`、`ToolStateChange`、`QuestionAsked`、`PermissionRequested`、`SessionIdle` 等类型化事件。
- `StreamingBridge` 负责本轮请求的事件监听、文本收集和最终响应。
- `AgentConsoleSession` 使用 CardKit streaming card 展示工具执行时间线。
- `InteractiveCardRegistry` 和 `InteractiveHandler` 处理问题、权限卡片的去重与回调。
- `CardKitClient` 封装卡片创建、元素更新、保活和关闭接口。

SRM Agent Hub 方案提供了值得参考的产品形态：同一张会话卡在 `streaming`、`question`、`permission`、`idle`、`closed` 等状态间演进，并在完成态提供继续对话入口。但其实际流式实现主要依赖反复 PATCH 整张消息卡片，CardKit streaming element 相关代码没有进入主路径。

本方案保留当前工程更成熟的 CardKit element 更新能力，吸收 SRM 的单卡会话体验，同时重点解决高频更新、队列积压、卡顿、伪动画、限流、超时回收和并发归属问题。

## 2. 设计目标

- 用户在 1 秒内看到明确的执行反馈。
- 工具状态和答案内容能够持续更新，但不按每个 SSE delta 请求飞书。
- 突发事件只发送最新状态，不排队发送已经过时的中间快照。
- 动效来自真实执行状态和 CardKit 客户端能力，不用服务端定时请求模拟 spinner。
- OpenCode 已完成后，最终结果在 2 秒内稳定可见。
- 单次 CardKit 失败、限流或网络抖动不会永久破坏后续更新。
- SSE 丢失 `SessionIdle` 时能够自动释放 listener、timer、reaction 和卡片资源。
- 问题和权限操作继续保持可靠、幂等和可追踪。
- 为后续“单主卡 + 卡内继续对话”保留演进空间。

## 3. 非目标

- 不直接展示完整 OpenCode 原始日志或 reasoning 内容。
- 不将每个 token 或 delta 直接发送给飞书。
- 不使用服务端高频 timer 制造逐帧 spinner。
- 第一轮优化不改变 OpenCode SSE 协议。
- 不在已有主卡的活跃会话中再发送独立问题或权限卡，避免重复交互入口。
- 不为兼容旧实现保留两套并行的生产流式链路。

## 4. 当前链路与问题

### 4.1 当前生产链路

```text
Feishu message
  -> MessageHandler
  -> StreamingBridge 注册 session listener
  -> POST /session/{id}/message
  -> OpenCode SSE
  -> EventProcessor
  -> AgentConsoleSession
       ToolStateChange -> CardKit 进度元素更新
        Question/Permission -> 主卡内嵌交互，必要时暂停/恢复 streaming
       TextDelta -> 内存 textBuffer
       SessionIdle -> 关闭进度卡 + 发送最终回复卡
```

### 4.2 核心问题

#### 答案不是真正流式

`TextDelta` 当前只累加到 `textBuffer`。用户看到的是工具进度卡，最终答案要等 `SessionIdle` 后通过另一张普通卡一次性发送。

#### 串行队列不等于节流

旧实现每个工具事件都生成一个完整 markdown 快照并加入 Promise 队列。如果 1 秒内出现 100 个状态事件，而 CardKit RTT 为 300ms，队列可能落后几十秒。中间快照即使已经过时，仍会逐条发送。

#### 队列可能永久失效

如果 Promise 队列中的一次 `updateElement()` 失败，队列尾部会保持 rejected，后续更新和 `close()` 都可能被跳过。

#### 服务端 spinner 成本高且没有业务价值

每 1 秒更新一次 spinner 意味着单个 session 每分钟约 60 次 CardKit 请求。多 session 并发时会与真实工具、答案更新竞争 sequence 和频率配额。动画帧在网络慢时还会积压，最终比静态状态更卡。

#### 生命周期回收不完整

只有首事件超时不足以覆盖以下情况：收到一个事件后 SSE 中断、OpenCode 卡住、`SessionIdle` 丢失、CardKit 关闭请求挂起。结果可能是 listener、reaction、heartbeat 和 busy 状态长期不释放。

#### 同 session 多请求可能串流

事件目前主要按 `sessionId` 分发。如果同一 session 同时存在两个请求，而处理后的事件没有保留 OpenCode `messageId`，两个 bridge listener 可能消费相同的文本和 idle 事件。

## 5. 目标用户体验

### 5.1 主卡状态

```text
starting
  -> running
  -> waiting_question / waiting_permission
  -> running
  -> completed / error / timeout
```

### 5.2 执行中

```text
飞码智能体

✓ 已读取 src/handler/streaming-integration.ts
✓ 已搜索 SessionManager
正在执行命令 bun test...
```

### 5.3 等待用户

```text
飞码智能体

✓ 已完成上下文分析
? 等待权限确认 bun test

请在当前卡片内选择。
```

问题和权限优先内嵌在主卡中。展示交互组件前暂停 CardKit streaming，插入 question 或 permission 组件；用户提交后删除交互组件并恢复 streaming。没有活跃主卡的场景仍保留独立操作卡，例如 TUI 主动触发的问题或权限请求。

### 5.4 完成态

近期保持“进度卡 + 最终回复卡”结构，优先保证稳定性。目标形态升级为一张主卡同时承载执行进度和答案：

```text
飞码智能体 · 已完成

执行进度
✓ 已读取 4 个文件
✓ 已修改 2 个文件
✓ 已通过测试

回复
最终完整答案……
```

## 6. 目标卡片结构

### 6.1 双元素主卡

目标主卡预置两个独立 markdown element：

| element_id | 职责 | 更新频率 |
| --- | --- | --- |
| `progress` | 工具、子任务、等待状态、错误摘要 | 真实状态变化时更新 |
| `answer` | 合并后的 OpenCode 文本输出 | 最多每 300ms 更新一次 |

CardKit `sequence` 属于整张卡，两个元素必须共用一个调度器，不能各自维护 sequence。

### 6.2 更新调度器

每张主卡维护一个 `CardUpdateScheduler`：

```text
latestProgress
latestAnswer
progressDirty
answerDirty
inFlight
sequence
lastSentProgress
lastSentAnswer
```

调度规则：

1. 事件只更新内存模型并标记 dirty。
2. 最小刷新间隔为 300ms。
3. 同时最多一个 CardKit 请求在途。
4. 请求期间新事件只覆盖 latest snapshot。
5. 在途请求结束后，只发送最新 pending snapshot。
6. 与上次成功内容相同则跳过。
7. `close()` 时优先 flush 最新答案和进度，但有明确时间预算。

示例：

```text
发送状态 1 期间：
状态 2 到达 -> pending = 状态 2
状态 3 到达 -> pending = 状态 3
状态 4 到达 -> pending = 状态 4

状态 1 完成 -> 直接发送状态 4
```

## 7. 流式内容策略

### 7.1 TextDelta 合并

```text
TextDelta
  -> append answerBuffer
  -> mark answer dirty
  -> scheduler
  -> 300ms 后发送最新答案快照
```

不能采用“一条 delta 对应一次 CardKit 请求”。

### 7.2 内容长度控制

飞书卡片大小限制和多字节字符要求决定了内容必须按 UTF-8 bytes 控制，而不是只看 JavaScript `string.length`。

建议：

- 流式答案区域保留 18KB 至 20KB 安全预算。
- 超长回答在卡片中显示截断提示。
- 完整内容通过后续普通消息、Markdown 文件或附件提供。
- 截断代码块时补齐 Markdown fence，避免整张卡渲染异常。
- 表格和超长单行应降级或截断。

### 7.3 短任务延迟建卡

建议延迟约 500ms 创建主卡：

- 500ms 内完成的简单问答直接发送最终回复，不创建短暂闪烁的进度卡。
- 出现工具、问题或权限事件时立即创建。
- 纯文本持续超过阈值时创建并展示答案。
- 建卡前发生的状态保存在内存模型中，建卡后一次渲染最新状态。

## 8. 动效设计

### 8.1 禁止服务端高频 spinner

不使用 `setInterval()` 每秒调用 CardKit 更新 spinner。原因：

- 没有新增业务信息。
- 占用单卡更新频率。
- 增加 429 风险。
- 与工具和答案更新竞争 sequence。
- 慢网络下动画帧会排队，产生反向卡顿。

### 8.2 真实状态动效

| 状态 | 表现 |
| --- | --- |
| starting | `正在分析请求...` |
| running | `正在读取 src/index.ts · 已用时 3 秒` |
| completed item | `✓ 已读取 src/index.ts · 用时 3.4 秒` |
| failed item | `! 读取失败 src/index.ts · 运行 3.4 秒` |
| waiting question | 黄色等待状态 + 主卡内嵌问题组件 |
| waiting permission | 黄色等待状态 + 主卡内嵌权限组件 |
| completed | 绿色完成状态 |
| error | 红色错误状态 + 简短原因 |

CardKit 客户端打印参数建议从以下值开始真机验证：

```text
print_frequency_ms: 70-100
print_step: 5-10
print_strategy: fast
```

进度区域不追求逐字符动画，目标是 1 秒内显示最新真实状态。

### 8.3 动作内联耗时

当前执行动作后直接追加弱化的耗时信息，使动作和运行时间保持在同一阅读上下文中：

```text
✓ 已读取 src/index.ts · 用时 1.8 秒
✓ 已搜索 StreamingBridge · 用时 4.2 秒
正在执行命令 bun test · 已用时 8 秒
```

展示规则：

- 只有 running item 的耗时持续变化。
- running 使用 `已用时`，completed 使用 `用时`，error 使用 `运行`。
- 完成或失败后冻结最终耗时，不再更新。
- 每个并行 running item 使用自己的 `startedAt` 计算耗时。
- 耗时紧跟动作文本，通过 ` · ` 分隔，不拆成独立行。
- 计时信息属于辅助信息，视觉权重应低于动作本身；如果 markdown 无法只将行尾设置为灰色，则保持同色并通过分隔符和简短文案弱化。
- 问题和权限等待使用 `已等待`，例如 `等待权限确认 · 已等待 2 分钟`，与工具执行耗时区分。

耗时格式：

| 时长 | 展示格式 |
| --- | --- |
| 小于 60 秒，运行中 | 整秒，例如 `已用时 8 秒` |
| 小于 60 秒，已结束 | 保留一位小数，例如 `用时 8.4 秒` |
| 1 至 60 分钟 | `2 分 30 秒` |
| 60 分钟以上 | `1 小时 12 分` |

### 8.4 自适应刷新频率

读秒不能在整个任务生命周期内固定每秒请求飞书。建议按当前 item 已运行时间降低刷新频率：

| 已运行时间 | 计时刷新频率 |
| --- | ---: |
| 0 至 10 秒 | 每 1 秒 |
| 10 至 60 秒 | 每 5 秒 |
| 1 至 10 分钟 | 每 30 秒 |
| 超过 10 分钟 | 每 1 分钟 |

调度约束：

- 计时 tick 只更新内存中的 elapsed 值并标记 progress dirty。
- 计时更新必须进入统一 `CardUpdateScheduler`，不能建立独立 Promise 队列。
- 工具状态、答案、问题、权限和完成事件的优先级高于计时更新。
- 请求在途时只保留最新计时值，中间秒数允许丢弃。
- 429、网络错误或调度拥塞时直接跳过装饰性计时帧，不为计时帧单独重试。
- 没有 running 或 waiting item 时停止计时器。
- `SessionIdle`、error、close 和所有 finalize 路径必须清理计时器。
- timer 应调用 `unref()`，不得阻止进程退出。
- 多个并行 running item 共用一个 session 级 timer，一次重渲染全部可见 item。

该方案保留前 10 秒明显的运行反馈，同时避免长任务持续产生每秒一次的 CardKit 请求。

## 9. HTTP 可靠性设计

### 9.1 超时与重试

CardKit 请求策略：

- 单请求超时：10 秒。
- 可重试：网络错误、HTTP 429、502、503、504。
- 最大重试：3 次。
- 优先使用 `Retry-After`。
- 否则采用退避：250ms、750ms、2000ms。
- 业务 4xx 不盲目重试。
- 同一次逻辑请求重试时复用原 `sequence`、`uuid` 和 body。
- 非 JSON 响应转换为结构化 `CardKitError`。

### 9.2 更新失败恢复

单次失败必须满足：

- 调用方可以观察错误并记录日志。
- 内部更新通道恢复为可用状态。
- 后续快照仍可发送。
- 最终 `closeStreaming()` 仍会 best-effort 执行。

## 10. 生命周期设计

### 10.1 超时分层

| 超时 | 默认值 | 作用 |
| --- | ---: | --- |
| 首事件超时 | 5 分钟 | 无 SSE 时使用同步响应回退 |
| 运行静默超时 | 15 分钟 | 收到事件后长期无活动时回收 |
| 等待用户静默超时 | 1 小时 | 问题或权限等待的长阈值 |
| 绝对生命周期 | 2 小时 | 防止任何请求永久占用资源 |
| 卡片关闭预算 | 3 秒 | 防止进度卡阻塞最终结果 |

每个有效 SSE 事件刷新活动时间。问题和权限等待态使用更长阈值，但仍受绝对生命周期限制。

### 10.2 统一完成流程

统一 `finalizeOnce()` 应负责：

- 标记请求 settled。
- 清理所有 timer。
- 移除 session listener。
- 停止卡片更新调度器和 heartbeat。
- 在预算内 flush/close 卡片。
- 删除 Typing reaction。
- 释放 `SessionObserver` busy 状态。
- 发送或更新最终答案。
- 处理输出文件。
- 释放 session 请求锁。

所有完成、错误、POST 失败、SSE 静默和绝对超时路径必须复用该流程。

## 11. 并发与事件归属

### 11.1 短期策略

在事件仍只能可靠关联到 `sessionId` 时，同一个 OpenCode session 同时只允许一个 active request。后续请求排队或明确提示当前会话忙碌，不允许两个 bridge listener 同时消费同一 session 的事件。

### 11.2 长期策略

扩展事件模型保留：

```ts
interface TextDelta {
  sessionId: string
  messageId: string
  partId?: string
  text: string
}
```

工具和子任务事件也尽可能保留 `messageId`、`partId`、`callId`。Bridge 在绑定本轮 assistant message 后，只处理属于当前 turn 的事件。完成该关联后再评估同 session 并行。

## 12. Timeline 模型

- 使用 `partId` 或 `callId` 作为工具调用稳定标识。
- 未提供 ID 时，completed/error 优先匹配最近一个同工具、同标题且仍 running 的 item。
- 使用 `Map<id, item>` 建立索引，避免每次 O(n) 查找。
- 只保留最近 100 至 500 条完整 item。
- 卡片默认展示最近 5 条。
- 更早步骤压缩成 `已折叠 N 个早期步骤`。
- 全局状态由 session 和 item 状态派生，后续 completed 事件不能覆盖已有 error 状态。

## 13. 问题、权限与命令交互

### 13.1 主卡内嵌优先

问题和权限交互采用“主卡内嵌优先、独立卡兜底”的策略：

- `StreamingBridge` 在活跃请求内维护主卡 session，并通过 `EmbeddedInteractionRegistry` 建立 `requestId -> 主卡交互` 映射。
- `AgentConsoleSession` 收到 `QuestionAsked` 或 `PermissionRequested` 后，先暂停 streaming，再插入交互组件。
- 用户提交成功后，服务端删除对应交互元素，恢复 streaming，并在主卡中保留冻结的等待耗时和用户选择摘要。
- `InteractivePoller` 遇到活跃主卡 session 时只记录 pending request，不再发送独立问题或权限卡。
- 如果当前没有活跃主卡，例如 TUI 主动触发的问题或权限请求，仍由独立卡承载交互。

该策略避免同一问题同时出现在主卡和独立卡中，也避免用户回答后因为轮询竞态再次看到相同问题。

### 13.2 CardKit streaming 与结构更新

CardKit 交互组件和 streaming 更新不能无序并发。主卡交互遵循以下顺序：

1. 注册 `requestId -> embedded interaction`。
2. 调用 `pauseStreaming()` 暂停流式打印。
3. 通过结构更新插入 question 或 permission 组件。
4. 用户提交后，`InteractiveHandler` 将答案或权限决定转发到 OpenCode。
5. OpenCode 接受后先取消注册，再删除交互元素。
6. 调用 `renewStreaming()` 恢复流式打印。

如果插入组件失败，会取消注册并记录日志。对于活跃主卡 session，不降级发送独立卡，优先避免重复交互入口。

### 13.3 问题多选体验

多选问题使用横向 `checker` Checkbox，而不是下拉式多选：

- 所有候选项横向排列。
- 确认按钮单独一行左对齐。
- 回调通过 `form_value.question_choice_N` 和 `value.optionLabels` 还原选中标签。
- 提交给 OpenCode 的格式为 `{"answers":[["选项一","选项二"]]}`。
- 回答成功后主卡保留 `你选择的是：**选项一，选项二**。`。

`requestId` 级幂等保护必须覆盖重复点击、重复 webhook 投递和主卡清理竞态。同一个问题只允许向 OpenCode 提交一次有效答案。

### 13.4 时间展示

问题和权限等待使用固定展示时刻：

- pending、running、completed 的 question 工具从首次工具事件绑定固定时刻。
- `QuestionAsked` 晚到不重置工具时刻。
- 等待用户回答行创建时绑定固定时刻。
- 用户回答后切换为冻结 `用时` 或 `已等待`。

初始等待阶段不展示动态 `已用时/已等待`，避免用户误以为已经等待很久。完成后再展示真实耗时。

### 13.5 命令菜单重复点击保护

Slash command 菜单点击后必须在飞书卡片回调响应中立即替换为不可交互文本，例如：

```text
已选择执行 **新建会话**。
```

后台 `updateMessage()` 仅作为兜底。服务端还需要按 `open_message_id` 做重复执行保护，防止用户或飞书重试导致同一菜单命令执行多次。

## 14. 完成态会话面板

吸收 SRM 方案的以下能力：

- 完成态卡片提供“继续对话”多行输入框。
- 提供“关闭会话”按钮。
- action 携带 `sessionId`，服务端校验操作者身份。
- 关闭后卡片变为灰色只读状态。
- 恢复操作必须先确认 OpenCode session 仍存在，不能只恢复本地状态。

权限规则至少限制为：session 发起人、当前 thread owner 或明确的群管理员。

## 16. 已实施基线

截至本方案更新，第一轮稳定性优化已经完成：

- `AgentConsoleSession` 使用 300ms latest-only 更新。
- 同时最多一个更新请求在途。
- pending 只保留最新内容。
- 更新失败后可继续更新和关闭。
- `start()` 支持 concurrent single-flight。
- 禁用服务端 spinner timer。
- CardKit `print_step` 从 1 调整为 5。
- CardKit 请求具备 10 秒超时和 429/502/503/504 重试。
- Bridge 具备运行静默、等待用户和绝对生命周期 watchdog。
- 卡片关闭具备 3 秒预算。
- 全量 463 个测试通过，TypeScript 构建通过。

阶段 2 已进一步完成：

- 主卡拆分为 `progress` 和 `answer` 两个 element，共享单调递增 sequence。
- `TextDelta` 经 300ms latest-only 调度器更新答案区域。
- 500ms 内完成的快速请求跳过进度卡，避免闪烁和多余 API 调用。
- 主卡成功完成时不再重复发送最终回复卡。
- 主卡失败或答案超过 18KB 时使用普通完整回复降级。
- 当前动作后内联展示自适应耗时，完成后冻结最终用时。
- 无 `partId` 的 completed/error 工具可以匹配最近的 running item。

阶段 3 已完成短期单飞保护：

- `StreamingBridge` 按 `sessionId` 维护 Promise gate。
- 同 session 后续请求在前一个请求完整 finalize 后才注册 listener 和执行 POST。
- 不同 session 仍可并行。
- gate 在正常完成、错误和 watchdog 路径通过 `finally` 释放。
- message 级事件关联仍待后续补齐；由于 `SessionIdle` 当前没有 turn ID，现阶段保留单飞作为正确性边界。

阶段 4 的主卡交互体验已提前部分落地：

- 问题和权限优先内嵌到活跃主卡，展示前暂停 streaming，提交后删除交互元素并恢复 streaming。
- 新增 `EmbeddedInteractionRegistry` 维护 `requestId -> 主卡交互` 映射。
- `InteractivePoller` 感知活跃主卡 session，避免为同一 request 再发送独立 question 或 permission 卡。
- 内嵌交互失败时，活跃主卡 session 不降级发送独立卡，优先避免重复交互入口。
- 多选问题使用横向 Checkbox，并通过独立确认按钮提交。
- 多选回调按 `optionLabels` 和 `question_choice_N` 解析，向 OpenCode 提交嵌套 answers 数组。
- question 回调具备 `requestId` 级幂等保护，重复 webhook 或重复点击只提交一次。
- 回答成功后主卡保留冻结的等待耗时和用户选择摘要。
- 命令菜单点击后通过 card callback response 立即替换为不可交互结果，并按 `open_message_id` 防重复执行。
- `Answered` toast 不再重复展示成功图标。
- “正在分析请求...”支持自适应读秒，首个答案或工具步骤出现后停止。
- question 工具和等待用户回答的初始展示使用固定时刻，用户回答后才冻结展示真实耗时。

上下文隔离与文本重建已补强：

- 新飞书 thread 默认创建干净 OpenCode session，不再自动继承最近 TUI session。
- 可通过 `session.autoDiscoverTui: true` 恢复旧发现行为；推荐使用 `/connect` 显式连接已有会话。
- 已有数据库映射不会自动删除，升级后旧话题需要执行一次 `/new` 才会切换到干净上下文。
- 显式连接的 session 如果正处于 busy/retry，飞书请求会被拒绝并提示稍后重试，不再消费旧 turn 输出。
- `TextDelta` 保留 `messageId`、`partId` 和 `source`，Bridge 只接收当前绑定 message 的文本。
- `EventProcessor` 处理 `message.updated` 并缓存 `messageId -> role`，明确丢弃 user message 的 text part；注入的 `[Lark context]` 只发送给 OpenCode，不进入飞书答案卡。
- `message.part.updated` 作为 snapshot，结合事件 delta 按最大文本重叠合并；`message.part.delta` 作为增量，避免双通道重复。
- 同一 part 的 snapshot 会校正已累积 delta，其他 messageID 的文本被丢弃。
- 首个非空 assistant 答案输出后，`progress` 中的“正在分析请求...”使用零宽占位更新清除，避免飞书忽略空字符串更新。
- 当前无工具答案直接复用 `progress` 元素承载结果，不再依赖零宽字符占位产生空白行。

## 17. 决策总结

1. 借鉴 SRM 的单卡会话体验，不照搬整卡高频 PATCH。
2. 保留 CardKit element 更新作为技术底座。
3. 所有更新使用 latest-only 合并，不保存过时快照队列。
4. 不使用服务端 spinner 动画。
5. 先保证稳定性，再升级为 `progress + answer` 双元素真流式主卡。
6. 问题和权限在活跃主卡中内嵌优先，无活跃主卡时才使用独立卡兜底。
7. 活跃主卡场景禁止独立交互卡降级，避免重复提问和重复授权入口。
8. 同 session 并发必须先解决 message 级事件归属。
9. 完成态继续对话和关闭会话作为后续会话面板能力实施。
