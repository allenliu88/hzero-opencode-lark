# 飞码智能体主卡片底部控制区方案设计

## 1. 背景

`opencode-lark` 使用 CardKit 2.0 streaming 主卡承载执行时间线、答案、问题与权限交互。主卡底部需要稳定展示本轮实际上下文，并提供无需调用模型的 slash command 帮助入口。

Agent、Model、Session 的切换不直接塞入 streaming 主卡，而是通过 `/agents`、`/models`、`/sessions` 打开独立的 Card JSON 2.0 分页选择卡。当前主卡只渲染只读上下文与帮助文本，没有直接打开选择器或中止任务的按钮。这样可以避免按钮文本截断、卡片元素过多和 streaming 卡复杂交互更新失败。

## 2. 当前用户体验

### 2.1 主卡标题

主卡创建前补齐 Session 与 VCS 信息，标题格式为：

```text
🧠 飞码智能体 · 项目名称#分支名称
```

信息来源：

- Session title、directory、当前模型：`GET /session/:sessionID`。
- 分支：`GET /vcs?directory=<absolute-directory>`。
- 项目名：mapping 中的 `project_name`，否则取 directory basename。

### 2.2 主卡底部

答案区域之后使用 `hr` 分隔，底部 markdown 为：

```text
**当前上下文**：会话 `会话名称` · `会话ID` · 项目 `项目名称#分支` · 智能体 `build` · 模型 `providerID/modelID`
**帮助**：`/agents` 切换智能体 · `/models` 切换模型 · `/sessions` 切换会话 · `/files` 浏览文件 · `/abort` 中止 · `/help` 帮助
```

模型必须是当前实际使用的模型，不使用模糊的“默认模型”占位。

实际模型初始化与刷新规则：

1. 主卡创建前使用 `GET /session/:sessionID` 的 `session.model` 初始化 mapping；Session 无模型时保留 SQLite mapping 中的 `provider_id/model_id`。
2. 本轮 User `message.updated` 的 `model.providerID/modelID` 和 Assistant `message.updated` 的 `providerID/modelID` 都可刷新模型。
3. 当前实现按符合条件的模型事件到达顺序覆盖；事件本身不携带额外优先级状态。
4. 全部缺失时显示 `未知模型`。

收到本轮实际模型后，`StreamingBridge` 写回 mapping，并通过 `AgentConsoleSession.setControls()` 更新 `agentCtlInfo` 元素。

## 3. 独立选择卡

### 3.1 共同规则

`/agents`、`/models`、`/sessions` 三类选择器统一由 `SelectionPickerRegistry` 管理。`AgentConsoleRegistry` 保留打开选择器的回调适配能力，但当前主卡未渲染对应入口：

- Card JSON 2.0，按钮 `width: fill`。
- 每页 8 行，与远程文件浏览器分页行为一致。
- 上一页/下一页在原卡更新，不发送新分页卡。
- 显示 `第 n/m 页`。
- 使用 `viewToken`、`actionId`、串行 action queue 和 TTL。
- callback 只携带 opaque `entryKey`，实际值从服务端 registry 解析。
- 只有创建卡片的用户可以操作。
- 过期、重复、越权和旧 view callback 返回明确 toast。

### 3.2 Agent 选择卡

```text
🤖 选择智能体

当前智能体：`build`
────────────────────────
▶ build
review
```

数据源：

```http
GET /agent?directory=<absolute-directory>
```

过滤规则与 OpenCode Web 一致：排除 `mode === "subagent"` 和 `hidden === true`，保留服务端顺序。

### 3.3 Model 选择卡

```text
🧩 选择模型

当前模型：`providerID/modelID`
────────────────────────
▶ providerID/modelID
```

数据源：

```http
GET /provider?directory=<absolute-directory>
```

OpenCode 返回 `connected` 数组时只展示其中的 provider；兼容旧响应时若该字段缺失，则保留所有可解析 provider。始终排除 `status === "deprecated"` 的模型。显示值固定为 `providerID/modelID`；内部选择值使用 `providerID:modelID`。

### 3.4 Session 选择卡

```text
📋 选择会话

当前会话：`会话名称` · `会话ID` · `项目名称#分支`
────────────────────────
▶ 会话名称 · 15分钟前 · 0文件
会话ID · 项目名称#分支
```

第二行为灰色小字。数据源：

```http
GET /session?directory=<absolute-directory>&roots=true&limit=10000
```

过滤规则：

- 只保留当前 directory。
- 只保留 root Session。
- 排除 archived Session。
- 当前 Session 置顶。
- 缺失文件数显示 `0文件`，缺失更新时间显示 `未知时间`。

未绑定 Session 时，directory 使用 `OPENCODE_CWD`，否则使用 `process.cwd()`。

## 4. OpenCode 控制适配层

`src/opencode/control-client.ts` 提供：

```ts
listAgents(directory?: string): Promise<AgentOption[]>
listModels(directory?: string): Promise<ModelOption[]>
listSessions(directory: string): Promise<SessionOption[]>
getSession(sessionId: string): Promise<SessionOption | null>
getVcs(directory: string): Promise<{ branch?: string; defaultBranch?: string }>
```

同时保留 `listProjects()`、`listProjectDirectories()`、`listProjectSessions()`、`createProjectSession()`、`abortSession()` 和 `getConsoleOptions()` 兼容方法。它们用于项目级 abort fallback 和尚未暴露的实验性项目切换，不作为 Agent、Model、Session 选择卡的主要数据源。当前输入 `/abort` 仍直接调用全局 `/session/:sessionID/abort`。

`SessionOption.model` 归一化为：

```ts
model?: {
  providerId: string
  modelId: string
}
```

## 5. Mapping 与发送语义

`SessionMapping` 持久化：

```text
session_id, session_title, project_id, directory, project_name, branch_name,
agent, provider_id, model_id
```

选择 Agent 或 Model 只更新当前 Feishu thread mapping，对下一条消息生效。`MessageHandler` 将 mapping 中的 agent/provider/model 加入 OpenCode message body；OpenCode 返回 HTTP 400 或 422 时，重试一次只包含 parts 的 body。该兼容重试会同时去掉 Agent 和 Model 覆盖，但不会清除持久化 selection，后续消息仍会再次尝试覆盖字段。

选择 Session 使用 `setMapping()` 替换当前 thread mapping，写入 session ID、title、directory、project name 和 branch。此前的 Agent、Model 和 project-ID 覆盖会重置；如需覆盖新 Session 的上下文，应再次使用 `/agents` 或 `/models`。

## 6. 安全与生命周期

- 选择器 callback 不信任客户端传回的业务 ID。
- message ID、chat ID、operator open ID、view token 和 action ID 必须全部匹配。
- registry 默认 30 分钟 TTL，服务重启后旧选择卡失效。
- 选择器 API 失败不影响主卡流式输出。
- 主卡完成态保留只读上下文与帮助行。
- 帮助卡 `command_execute` 回调目前只以 chat ID 作为 best-effort mapping key；p2p 行为准确，群话题中建议直接发送 slash command，以保留完整 thread key。
- 命令菜单按 card message ID 做进程内重复抑制；该状态不持久化、当前不限制为卡片创建者，且进程生命周期内不会自动过期。

## 7. 验收标准

- 主卡标题包含项目和分支。
- 主卡底部显示会话、项目分支、Agent 和实际 `providerID/modelID`。
- 帮助行包含 `/agents`、`/models`、`/sessions`、`/files`、`/abort`、`/help`。
- 三类选择器使用相同的 8 行原卡分页机制。
- Agent、Model 和 Session 数据源与 OpenCode Web 一致。
- 当前项置顶或使用 `▶` 标记。
- 全量构建、测试、diff check 和 ast-grep 检查通过。
